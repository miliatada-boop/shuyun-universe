# 项目维护模式 Skill (Project Maintenance Mode Skill)

## 核心概念

这是一个**双层级维护模式**，融入"双重检查（Double Check）"工作习惯。

### 层级一：单项目维护模式（Deep Maintenance）
- 针对单个项目的深度检查
- 在项目收尾时或工作结束前运行
- 检查内容更深入、更细致

### 层级二：全局维护模式（Global Maintenance）
- 批量检查所有当日改动的项目
- 适用于多项目并行工作（一天 7-8 个窗口）
- 快速了解各项目状态和需要处理的问题

---

## 何时使用

**单项目维护**：
- ✅ 单个项目快要收尾时
- ✅ 工作日结束前，对当前项目做最后的双重检查
- ✅ 需要确保没有遗漏的改动或待提交的内容

**全局维护**：
- ✅ 多项目并行工作日（一天打开多个项目窗口）
- ✅ 工作日快结束时，需要了解所有项目的状态
- ✅ 快速审查各项目的增减变化

---

## 维护工作流程

### 阶段 1️⃣：项目扫描 & 版本状态提取

**单项目模式**：
```bash
# 扫描项目状态
git status
git log --oneline -20

# 提取版本信息（前 20 行）
head -20 CHANGELOG.md 或 package.json 中的 version
```

**全局模式**：
```bash
# 扫描所有当日改动的项目
find ~ -type d -name ".git" -mtime -1 2>/dev/null | head -10

# 每个项目提取版本信息
for project in $(已改动项目列表); do
  echo "=== $project ===" 
  head -20 $project/CHANGELOG.md
done
```

**记忆对比**：
与项目 Memory 记录对比，记录：
- 🆕 新增文件/功能
- ✏️ 修改的内容
- ❌ 删除的文件/功能

---

### 阶段 2️⃣：并行检查（多维度审计）

#### A. 基础设施与日志检查
```bash
# 检查基础设施配置
- Docker/K8s 配置文件是否最新
- 环境变量是否完整
- 日志输出是否正确
- 定时任务是否配置
```

#### B. 一致性检查
```bash
npm run lint
npx tsc --noEmit
eslint src/ --max-warnings 0
```

#### C. 防膨胀扫描（冗余代码和文件）
```bash
- 未使用的导入
- 死代码（函数/变量无人引用）
- 重复的文件或代码
- 过大的文件（>500 行）
- 可删除的测试文件
```

#### D. 部署冗余扫描
```bash
- 构建产物大小
- 依赖是否有多余项
- 是否存在测试代码被打包进去
```

#### E. Cloud 健康度检查
```bash
# 如果部署到云端
- API 端点是否正常
- 数据库连接
- 缓存服务状态
- 错误日志是否异常
```

#### F. 代码审查
```bash
- console.log 等调试代码
- TODO/FIXME 注释
- 类型错误
- 潜在的 Bug
- 安全问题（硬编码密钥等）
```

#### G. 测试覆盖检查
```bash
npm run test -- --coverage

检查：
- 测试通过率
- 覆盖率目标
- 孤立的测试文件
- E2E 测试状态
```

---

### 阶段 3️⃣：记忆沉淀

记录以下内容到项目 Memory：
- 📝 本次维护的时间
- 📊 发现的问题数量和类型
- 🔧 已修复的内容
- ⏳ 待后续处理的事项
- 📈 项目健康度评分

## 最佳实践

### ✅ 何时运行维护模式

| 场景 | 推荐频率 | 维护类型 |
|------|---------|---------|
| 单项目开发完成 | 1次/项目 | 深度维护 |
| 工作日结束 | 每天 1次 | 全局维护 |
| 版本发布前 | 1次/版本 | 深度维护 |
| 大规模重构后 | 1次 | 深度维护 |
| 多项目并行 | 每 2小时 1次 | 全局维护 |

### 📌 双重检查（Double Check）习惯

1. **开发过程中**：边写代码边检查
2. **功能完成后**：第一遍检查（本地测试）
3. **工作快结束时**：第二遍检查（运行维护模式）
4. **提交前**：第三遍检查（代码审查）

### 🧠 记忆沉淀的作用

- 追踪项目成长轨迹
- 快速定位问题根源
- 避免重复同样的错误
- 为后续迭代提供参考

├─ 检查是否有未提交改动
├─ 提醒还有哪些改动需要提交

## 总结

**项目维护模式的本质**：
- 🎯 系统化的质量检查
- 📝 完整的问题记录
- 💾 有意义的改动追踪
- 🧠 知识的沉淀和复用

**关键一句话**：
> "维护模式不是结束工作，而是为下一次工作打基础。"
